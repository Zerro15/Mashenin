# AGENTS

## Назначение

Этот файл задает правила работы для агента в репозитории `mashenin`.
Цель правил: двигать проект как продуктовый мессенджер маленькими, проверяемыми шагами без лишнего шума в кодовой базе.

## Базовый режим работы

Агент работает по схеме:

1. Сначала аудит текущего состояния.
2. Потом короткий и конкретный план.
3. Потом минимальные изменения в пределах задачи.
4. Потом короткий отчет по результату.

Нельзя перескакивать сразу к большим правкам без понимания текущего состояния кода и контрактов.

## Правила внесения изменений

- Сначала понять текущее состояние кода, а не придумывать архитектуру с нуля.
- Работать минимальными и локальными правками.
- Не делать лишний рефакторинг.
- Не трогать несвязанные части проекта.
- Не исправлять попутно все найденные проблемы, если они не входят в текущий scope.
- Не удалять legacy-код без прямой задачи.
- Если задача большая, не расползаться по всему проекту, а ограничивать scope.
- Если задача требует заметного объема работ, сначала предложить небольшой vertical slice.

## Правила коммуникации

Перед изменениями агент обязан явно указать:

- какие файлы будут изменены;
- что именно планируется сделать в этих файлах;
- какие части проекта сознательно не затрагиваются.

После изменений агент обязан показать:

- какие файлы изменены;
- что сделано;
- как проверить локально;
- какие риски остались.

Если задача не помещается в разумный небольшой шаг, агент должен:

- зафиксировать границу текущего шага;
- предложить следующий маленький slice;
- не продолжать бесконтрольно расширять работу.

## Текущее направление проекта

На текущем этапе проект развивается как мессенджер для обычного пользователя.

Принятые ориентиры:

- основной frontend: `Next.js` в `apps/web`;
- основной backend: `Fastify` в `apps/api`;
- `apps/web/src/server.js` использовать только как reference;
- `apps/api/src/lib/router.js` использовать только как reference;
- разработка идет маленькими vertical slice;
- первый основной slice: `login -> open room -> load message history -> send message`.

## Что нельзя делать без отдельной задачи

- менять продуктовый scope без явного решения;
- делать большой архитектурный рефакторинг;
- удалять legacy-части;
- менять сразу несколько независимых подсистем в одной задаче;
- добавлять временные хаки без явной фиксации в итоговом отчете;
- смешивать документационные, продуктовые и инфраструктурные изменения без причины.

## Git-правила

Агент обязан соблюдать следующие правила:

- Никогда не коммитить напрямую в `main`.
- Перед началом задачи проверять текущую ветку и `git status`.
- Для каждой задачи создавать отдельную ветку.
- Использовать имена веток:
  - `feat/...`
  - `fix/...`
  - `refactor/...`
  - `docs/...`
  - `test/...`
- Коммиты делать маленькими и атомарными.
- Перед коммитом обязательно показывать `git status --short`.
- Не смешивать несвязанные изменения в один коммит.
- Если появляются временные или generated файлы, не включать их без причины.
- После изменений предлагать подходящее имя ветки и `commit message`.
- После завершения slice предлагать краткий итог и следующий шаг.

## Практика принятия scope

Если задача касается только одного пользовательского сценария, агент должен держаться именно этого сценария.

Пример правильного поведения:

- если задача про `login -> room -> messages`, не трогать `voice`, `events`, `friends`, `communities`;
- если задача про организацию репозитория, не менять код продукта;
- если задача про один endpoint, не переписывать весь слой API.

## Ожидаемый формат результата

Хороший результат для одной задачи:

- понятен по scope;
- ограничен по затронутым файлам;
- проверяем локально;
- не ломает соседние части проекта;
- оставляет ясный следующий шаг.

## Контекст проекта

- проект называется **Mashenin**
- основная рабочая ветка сейчас: `fix/livekit-cloud-runtime`
- стек:
  - web: `Next.js` в `apps/web`
  - api: `Fastify` в `apps/api`
  - browser tests: Playwright / headless Chromium
- локальные адреса:
  - web: `http://127.0.0.1:3001`
  - api: `http://127.0.0.1:4000`
- legacy reference (не использовать как основной runtime):
  - `apps/web/src/server.js`
  - `apps/api/src/lib/router.js`

## Стиль работы

- работать строго маленькими узкими slice;
- один slice = одна конкретная цель;
- не смешивать несколько задач в одном изменении;
- сначала подтверждать текущее поведение, потом менять код;
- если баг не подтверждён — не придумывать фикс;
- отвечать и писать отчёты только на русском.

## Обязательный workflow для каждого slice

Всегда:

1. показать `git branch --show-current`
2. показать `git status --short`
3. сделать короткий аудит только релевантных файлов
4. только потом делать тест или изменение
5. после изменения:
   - показать точный diff только изменённых файлов
   - прогнать релевантную проверку
   - снова показать `git status --short`

## Правила изменений

- не трогать voice/livekit/friends/events/invite, если slice не про них;
- не трогать infra, docker, cloudflare, nginx, tunnel без прямой необходимости;
- не менять сразу много файлов без причины;
- предпочитать минимальный фикс;
- не делать рефакторинг «заодно»;
- не добавлять новые зависимости без необходимости.

## Правила browser-level тестов

- для UI-проверок использовать только реальный Playwright browser test;
- не подменять browser-level проверку curl-эмуляцией;
- если тест идёт через Playwright, явно писать:
  - какой URL проверялся
  - какой сценарий был пройден
  - что именно подтверждено
- временные тестовые файлы хранить вне репозитория, например в `/tmp/*.mjs`

## Формат финального отчёта

Всегда использовать ровно такой формат:

1. Статус
2. Точная причина
3. Что изменено
4. Как проверено
5. Что теперь работает
6. Что ещё не закрыто
7. Какие файлы изменены
8. Какой commit сделать

## Git hygiene

- не коммитить без явного запроса пользователя;
- перед предложением коммита проверить, что в diff нет мусора;
- generated/temp/state/cache файлы не должны попадать в git;
- учитывать, что такие вещи должны игнорироваться:
  - `.qwen/`
  - `.next/`
  - `.next_backup/`
  - `.next_stale_*/`
  - runtime state (`data/*.json`)
  - credentials
  - temp browser test files

## Исторический контекст

### Подтверждённые browser-level сценарии

- **login → open room → send message → message persists after refresh → second user sees message**
  - проверено через Playwright Chromium на `127.0.0.1:3001`;
  - регистрация через API, login через UI форму, textarea + submit button в DOM.

- **logout flow через Header**
  - кнопка «Выйти» очищает сессию и редиректит на `/login`;
  - stale user name исчезает из Header;
  - без логина `/room/general` редиректит на `/login`.

- **edge cases формы отправки**
  - пустое сообщение → кнопка disabled;
  - whitespace-only → кнопка disabled;
  - длинный текст (5000 символов) → отправляется, текст в DOM;
  - спецсимволы (кириллица, эмодзи, кавычки, `<tag>`) → корректно в DOM;
  - Enter не отправляет, Shift+Enter добавляет перенос строки;
  - быстрый повторный клик → кнопка блокируется, дублей нет.

### Найденные и исправленные баги

- **stale auth state на room page** (`apps/web/pages/room/[roomId].tsx`)
  - polling добавлена проверка `/api/auth/me` перед каждым циклом;
  - при 401 → `clearSessionToken()` + redirect на `/login?next=/room/<id>`;
  - пользователь автоматически возвращается в комнату после ре-логина.

- **next path для dynamic route** (`apps/web/lib/session.ts`)
  - `router.asPath` возвращал шаблон `/room/[roomId]` вместо `/room/general`;
  - заменено на `window.location.pathname` — фактический путь браузера;
  - logout → re-login теперь возвращает в `/room/general`, а не в 404.
