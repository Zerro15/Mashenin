# AGENTS

## Назначение

Этот файл задает рабочий контракт для Codex CLI в репозитории `mashenin`.
Цель: вести проект маленькими проверяемыми slice, не терять контекст между сессиями и безопасно использовать subagents.

## Главные источники истины

При конфликте документов использовать такой порядок:

1. `AGENTS.md` — правила работы агента и subagents.
2. `PROJECT_STATE.md` — текущее состояние проекта и active scope.
3. `BACKLOG.md` — ближайшие slice-задачи.
4. `ARCHITECTURE.md` — краткая карта active runtime и границы legacy.
5. `DECISIONS.md` — уже принятые решения.

`README.md`, `PRODUCT_VISION.md`, `GIT_WORKFLOW.md`, `docs/*`, `CLAUDE.md`, `GITHUB_SETUP.md` не должны переопределять этот слой.

## Базовый режим работы

Всегда работать по схеме:

1. Проверить текущую ветку и `git status --short`.
2. Прочитать актуальный контекст из `AGENTS.md`, `PROJECT_STATE.md`, `BACKLOG.md`, при необходимости последние записи в `SESSION_LOG.md`.
3. Сделать короткий аудит только релевантных файлов.
4. Сформулировать маленький конкретный slice.
5. Выполнить минимальные изменения в пределах этого slice.
6. Прогнать релевантную проверку.
7. Обновить project memory, если состояние проекта реально изменилось.
8. Дать короткий итог, риски и следующий шаг.

Нельзя перескакивать сразу к большим изменениям без понимания текущего поведения и контрактов.

## Scope

- Один slice = одна конкретная цель.
- Не делать большой рефакторинг без отдельной задачи.
- Не трогать несвязанные части проекта.
- Не исправлять попутно все найденные проблемы.
- Не удалять legacy-код без прямого решения.
- Если задача большая, сначала выделять узкий vertical slice.

## Активный runtime и границы

- Основной frontend: `apps/web` на `Next.js`.
- Основной backend: `apps/api` на `Fastify`.
- `apps/web/src/*` и `apps/api/src/lib/router.js` считать reference-слоем, а не основным runtime.
- Текущий главный пользовательский путь: `login -> open room -> load message history -> send message`.

Без отдельной задачи не трогать:

- `voice`, `livekit`, `friends`, `events`, `invite`, если slice не про них;
- `infra`, `infrastructure`, `docker`, `cloudflare`, `nginx`, `tunnel`;
- несколько независимых подсистем в одной задаче.

## Правила коммуникации

Перед изменениями агент обязан явно указать:

- какие файлы будут изменены;
- что именно планируется сделать;
- какие части проекта сознательно не затрагиваются.

После изменений агент обязан показать:

- какие файлы изменены;
- что сделано;
- как проверить локально;
- какие риски остались.

Если задача не помещается в разумный шаг, нужно зафиксировать границу текущего slice и предложить следующий маленький шаг.

## Subagents

Subagents использовать только если это реально ускоряет работу и write scope можно разделить без конфликтов.

### Что делает только main agent

- определяет scope текущего slice;
- выбирает, нужны ли subagents;
- раздает ownership по файлам;
- интегрирует результаты;
- запускает финальные проверки;
- обновляет `PROJECT_STATE.md`, `BACKLOG.md`, `SESSION_LOG.md`, `ARCHITECTURE.md`;
- готовит итоговый отчет и предложение по commit.

### Допустимые роли

- `explorer` — read-only аудит кода или документов;
- `worker-web` — изменения только в явно выделенных файлах `apps/web`;
- `worker-api` — изменения только в явно выделенных файлах `apps/api`;
- `worker-test` — изменения только в тестах или временных harness-файлах.

### Правила для subagents

- Не выдавать двум writer-agents пересекающиеся write sets.
- Не давать subagent одновременно менять код и project memory.
- Не давать subagent менять `AGENTS.md`, `PROJECT_STATE.md`, `BACKLOG.md`, `SESSION_LOG.md`, `ARCHITECTURE.md`; это делает только main agent.
- Не отправлять subagents в reference-слой без прямой причины.
- Для hotspots использовать только одного writer-agent за раз.

Hotspots повышенного риска:

- `apps/web/pages/room/[roomId].tsx`
- `apps/web/lib/session.ts`
- `apps/api/src/lib/store.js`
- `apps/api/src/routes/auth.js`
- `apps/api/src/routes/rooms.js`

## Обязательный workflow для каждого slice

Всегда:

1. показать `git branch --show-current`
2. показать `git status --short`
3. сделать короткий аудит только релевантных файлов
4. только потом делать тест или изменение
5. после изменения:
   - показать точный diff только измененных файлов
   - прогнать релевантную проверку
   - снова показать `git status --short`

## Правила проверок

- Для UI-проверок использовать реальный browser-level сценарий, если задача именно про UI.
- Не подменять browser-level проверку curl-эмуляцией.
- Для текстового MVP сначала предпочитать релевантные smoke/e2e-проверки active runtime.
- Временные test-файлы хранить вне репозитория, например в `/tmp/*.mjs`.

## Как обновлять project memory

После каждого значимого завершенного slice:

- `PROJECT_STATE.md` обновлять только если реально изменилось состояние проекта, active flow или ограничения;
- `BACKLOG.md` обновлять, если изменился приоритет, статус или следующий slice;
- `SESSION_LOG.md` дополнять короткой новой записью;
- `ARCHITECTURE.md` менять только если изменился runtime map или границы active/reference слоя.

Нельзя превращать эти файлы в длинные дневники или дублировать в них одно и то же.

## Git-правила

- Не коммитить напрямую в `main`.
- Перед началом задачи всегда проверять текущую ветку и `git status --short`.
- Для каждой задачи использовать отдельную ветку:
  - `feat/...`
  - `fix/...`
  - `refactor/...`
  - `docs/...`
  - `test/...`
- Не коммитить без явного запроса пользователя.
- Перед предложением коммита проверять, что в diff нет мусора.
- Generated, temp, state и credentials не должны попадать в git.

## Формат финального отчета

Всегда использовать ровно такой формат:

1. Статус
2. Точная причина
3. Что изменено
4. Как проверено
5. Что теперь работает
6. Что еще не закрыто
7. Какие файлы изменены
8. Какой commit сделать
