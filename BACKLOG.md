# BACKLOG

## Уже есть

- Монорепо с основными каталогами: `apps/web`, `apps/api`, `packages/shared`, `infra`, `docs`.
- Основной frontend выбран как `Next.js` в `apps/web`.
- Основной backend выбран как `Fastify` в `apps/api`.
- Базовая схема данных для пользователей, комнат, сообщений, сессий и событий.
- `file`- и `sql`-провайдеры хранения.
- Базовые backend-операции для auth, комнат, сообщений и событий.
- Legacy SSR web и legacy router сохранены как reference.
- Понимание первого приоритетного vertical slice: `login -> open room -> load message history -> send message`.

## Реализовано (текущая сессия)

- Real-time чат через WebSocket (`/ws/rooms/:roomId` с auth).
- Typing indicator — кто печатает в комнате.
- Online presence — кто подключён к комнате через WS.
- Счётчик непрочитанных на каждой комнате и в Header.
- Browser notifications при новых сообщениях (когда вкладка не активна).
- Auto mark-as-read при входе в комнату.
- Список участников комнаты с online/offline статусом.
- Редактирование и удаление сообщений (только автор).
- Контекстное меню сообщений с outside-click close.
- Авто-скролл к последнему сообщению (отключается при скролле вверх).
- Индикатор статуса подключения (онлайн/подключаюсь/офлайн).
- Offline banner при потере WS подключения.
- Страница /settings — профиль, смена имени и about.
- HTTP polling как fallback когда WS недоступен.

## Текущий приоритет

- Свести Next frontend и Fastify API в рабочий текстовый сценарий мессенджера.
- Зафиксировать и удерживать минимальный продуктовый контракт для:
  - login;
  - список комнат;
  - открытие комнаты;
  - загрузка истории;
  - отправка сообщения.
- Убедиться, что этот сценарий можно проверить локально без захода в voice и второстепенные сущности.
- Снизить расхождения между основным стеком и legacy reference-кодом.

## MVP

- Стабильный вход пользователя.
- Понятный список комнат.
- Открытие комнаты по id или slug.
- История сообщений в комнате.
- Отправка нового сообщения.
- Минимальная обработка ошибок в UI.
- Единый контракт ответа API для основного сценария.
- Базовые smoke/e2e-проверки для основного пути пользователя.
- Прозрачная локальная проверка через `apps/web` и `apps/api`.

## Потом

- **Voice / LiveKit** — стабильное подключение, UI звонка, mute/unmute.
- Invite-based onboarding в продуктово завершенном виде.
- Отдельный slice `create hub / create room shell`.
- События и RSVP.
- Поиск по сообщениям.
- Прикрепление файлов к сообщениям.
- Мобильная адаптация интерфейса.

## Не трогаем пока

- Полноценный `voice / LiveKit`-сценарий.
- `events` как следующий обязательный slice.
- `friends`.
- `communities`.
- Большой cleanup legacy-кода.
- Масштабный рефакторинг frontend-архитектуры.
- Масштабный рефакторинг backend-архитектуры.
- Переезд на новую инфраструктуру без необходимости.
