# DECISIONS

## Назначение файла

Здесь фиксируются актуальные архитектурные и продуктовые решения, чтобы не обсуждать заново уже принятые базовые направления.

## Текущие решения

### 1. Основной frontend: Next.js

Принято решение, что основной frontend проекта развивается в `apps/web` на `Next.js`.

Следствие:

- новые продуктовые экраны и сценарии делаем в `apps/web`;
- новые пользовательские vertical slice доводим именно в этой реализации;
- параллельные UI-реализации не считаются основным направлением.

### 2. Основной backend: Fastify

Принято решение, что основной backend проекта развивается в `apps/api` на `Fastify`.

Следствие:

- новые API-изменения и стабилизация контрактов делаются здесь;
- именно Fastify-маршруты считаются основной реализацией backend-логики;
- reference-код не должен определять основное поведение проекта.

### 3. Legacy SSR web использовать только как reference

Файл `apps/web/src/server.js` и связанный с ним legacy SSR web не удаляются сейчас, но не используются как основная продуктовая ветка.

Следствие:

- можно смотреть в legacy web как в reference;
- нельзя развивать продукт, опираясь на него как на основной runtime;
- cleanup этой части откладывается до отдельной задачи.

### 4. Legacy router использовать только как reference

Файл `apps/api/src/lib/router.js` сохраняется как reference, но не рассматривается как основная backend-реализация.

Следствие:

- можно использовать его для понимания старых контрактов;
- нельзя делать его центром новых изменений;
- cleanup этой части откладывается до отдельной задачи.

### 5. Сначала доводим текстовый сценарий мессенджера

Текущий приоритет развития:

- login;
- open room;
- load message history;
- send message.

Причина:

- это минимальный проверяемый сценарий настоящего мессенджера;
- он дает основу для дальнейших функций;
- он проще и дешевле для стабилизации, чем voice-first сценарий.

### 6. Voice / LiveKit позже

Интеграция voice и LiveKit сохраняется в архитектуре проекта, но не является текущим центром разработки.

Следствие:

- voice не блокирует доведение текстового MVP;
- ошибки и пробелы в voice-части сейчас не должны расширять scope текстовых задач;
- к voice возвращаемся после стабилизации базового текстового контура.

### 7. Работаем маленькими проверяемыми шагами

Принято решение вести разработку через небольшие vertical slice.

Следствие:

- не делать большой рефакторинг без отдельной причины;
- не объединять несколько независимых направлений в один шаг;
- каждую задачу завершать понятным локальным результатом;
- после каждого шага фиксировать следующий маленький slice.

## Как обновлять этот файл

Новые решения добавляются только если:

- они реально влияют на направление разработки;
- они уже согласованы;
- команда без фиксации будет возвращаться к тем же обсуждениям.

Не нужно записывать сюда мелкие тактические детали, которые относятся только к одной локальной правке.
