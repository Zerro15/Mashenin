# GIT WORKFLOW

## Цель

Git-процесс в `mashenin` должен помогать вести разработку маленькими и понятными шагами, а не засорять историю случайными смешанными изменениями.

## Базовые принципы

- Никогда не работать с коммитами напрямую в `main`.
- Каждая задача идет в отдельной ветке.
- Один логический шаг — один небольшой набор связанных изменений.
- История должна быть читаемой и объяснимой.
- В коммит не должны попадать случайные временные файлы.

## Перед началом задачи

Перед началом работы нужно всегда проверить:

1. текущую ветку;
2. `git status --short`;
3. нет ли в дереве несвязанных изменений, которые могут испортить scope задачи.

Если рабочее дерево уже грязное:

- не трогать чужие изменения;
- не смешивать их со своей задачей;
- при необходимости коммитить только явно выбранные файлы.

## Как создавать ветки

Для каждой задачи создается отдельная ветка от актуальной рабочей базы.

Типы веток:

- `feat/...` — новая пользовательская или продуктовая функциональность;
- `fix/...` — исправление конкретной ошибки;
- `refactor/...` — ограниченный рефакторинг без смены продуктового поведения;
- `docs/...` — документация, правила работы, организационные изменения;
- `test/...` — тесты и проверочные сценарии.

## Как называть ветки

Имя ветки должно быть коротким и предметным.

Хорошие примеры:

- `feat/message-room-slice`
- `fix/room-messages-contract`
- `docs/project-foundation`
- `test/login-room-flow`
- `refactor/api-room-handler`

Плохие примеры:

- `work`
- `temp`
- `stuff`
- `new-branch`
- `fix-everything`

## Как оформлять коммиты

Коммит должен быть:

- небольшим;
- атомарным;
- понятным без чтения всей истории задачи.

Хороший commit message:

- описывает суть изменения;
- не слишком общий;
- не смешивает несколько независимых тем.

Хорошие примеры:

- `docs: add project workflow and product documents`
- `feat: add room page for message history`
- `fix: align room messages api contract`
- `test: add smoke checks for login to room flow`

Плохие примеры:

- `fix`
- `update`
- `changes`
- `wip`
- `misc`

## Когда делать отдельный commit

Нужно делать отдельный commit, если:

- завершен один логический шаг;
- следующая правка уже про другую часть задачи;
- можно отдельно проверить текущий результат;
- изменения уже можно ревьюить сами по себе.

Не нужно складывать в один commit:

- документацию и несвязанный код;
- refactor и новую фичу, если их можно разделить;
- generated-файлы и реальные изменения, если generated не обязателен.

## Как не засорять историю

Чтобы не засорять историю:

- не коммитить временные файлы;
- не коммитить build-артефакты без причины;
- не включать локальные backup-каталоги;
- не коммитить случайные форматирования в несвязанных файлах;
- не смешивать чужие незавершенные изменения со своей задачей.

Перед каждым коммитом обязательно посмотреть:

```bash
git status --short
```

И убедиться, что staged-изменения относятся только к текущей задаче.

## Что делать после завершения задачи

После завершения каждого slice нужно:

1. коротко зафиксировать, что сделано;
2. перечислить, как это проверить локально;
3. назвать оставшиеся риски;
4. предложить следующий маленький шаг;
5. предложить подходящее имя ветки и commit message, если они еще не созданы.

Если задача затрагивает текущий MVP-контур `login -> room -> messages -> logout`, перед commit нужно отдельно прогонять:

```bash
cd apps/api
npm run test:smoke
```

## Рекомендуемый практический сценарий

Пример обычного процесса:

1. Проверить текущую ветку и `git status --short`.
2. Создать новую ветку нужного типа.
3. Выполнить только один ограниченный scope.
4. Проверить локально.
5. Снова показать `git status --short`.
6. Закоммитить только относящиеся к задаче файлы.
7. Описать итог и предложить следующий slice.
