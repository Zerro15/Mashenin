# PRODUCT VISION

## Что такое Mashenin

Mashenin — это удобный мессенджер для обычного пользователя, в котором общение строится вокруг постоянных комнат, быстрых входов в диалог и простого сценария использования без перегруженного интерфейса.

Продукт должен помогать человеку быстро:

- войти в аккаунт;
- открыть нужную комнату;
- увидеть текущий контекст;
- прочитать историю;
- отправить сообщение;
- позже, при развитии продукта, подключиться к голосовому общению.

## Для кого продукт

Основная аудитория:

- небольшие команды;
- дружеские сообщества;
- постоянные группы общения;
- пользователи, которым нужен понятный и спокойный интерфейс без лишней сложности.

Mashenin не ориентирован на массовую публичную социальную сеть или тяжелый enterprise-комбайн.

## Чем Mashenin отличается

Mashenin делает ставку на несколько принципов:

- простые и постоянные комнаты вместо сложной иерархии;
- быстрый вход в нужный контекст;
- минимальный когнитивный шум;
- понятный сценарий для обычного пользователя;
- возможность постепенно дорасти до voice-first модели, не ломая базовый текстовый сценарий.

Сейчас особенно важно не перегружать продукт второстепенными функциями раньше времени.

## Выбранный путь развития

Текущий путь развития:

- основной frontend: `Next.js` в `apps/web`;
- основной backend: `Fastify` в `apps/api`;
- развитие идет маленькими vertical slice;
- сначала доводим базовый текстовый пользовательский сценарий;
- voice и более сложные real-time части добавляем позже, когда текстовый контур станет надежным.

Это означает, что проект сейчас развивается не через большой рефакторинг, а через короткие проверяемые шаги.

## Ближайший MVP

В ближайший MVP входят:

- вход пользователя;
- список комнат;
- открытие комнаты;
- загрузка истории сообщений;
- отправка сообщения;
- базовая модель пользователя, комнаты и сообщения;
- минимально стабильные продуктовые контракты между frontend и backend.

После этого можно расширять MVP в сторону:

- присутствия;
- invite/onboarding;
- событий;
- голосового сценария.

## Что сознательно не делаем пока

На текущем этапе сознательно не ставим в приоритет:

- полный voice-сценарий;
- сложные community-функции;
- публичное discovery;
- тяжелую модерацию;
- мобильные приложения;
- большой cleanup legacy-кода;
- масштабный архитектурный рефакторинг;
- украшательский UI ради UI без завершения базового пользовательского пути.

## Критерий правильного движения

Правильное развитие Mashenin сейчас выглядит так:

- каждый следующий шаг улучшает реальный пользовательский сценарий;
- изменения небольшие и проверяемые;
- продукт становится понятнее, а не сложнее;
- кодовая база не засоряется лишними побочными изменениями;
- каждое решение уменьшает разрыв между выбранным стеком и реальным поведением продукта.
